Abstract

Potholes are a persistent infrastructure and road-safety problem that causes vehicle damage, increases accident risk, and raises municipal maintenance costs. Traditional reporting methods, including manual surveys and complaint-based workflows, are often delayed and inconsistent. This project addresses that operational gap by providing an end-to-end pothole reporting and management system with an AI detection engine, production-oriented backend, geospatial database, Android field client, and web dashboard.

The implemented platform receives camera frames on Android devices, performs on-device pothole detection using a YOLOv8n model exported to TensorFlow Lite float16, and converts high-confidence detections into geotagged reports. Backend services validate and deduplicate these reports using spatial logic, store records in PostgreSQL/PostGIS, and expose secured APIs for maintenance operations. The dashboard supports map-based visualization, filterable lists, detailed record editing, status workflow updates, bulk actions, vehicle management, user management, CSV export, and analytics summaries. The system is containerized through Docker Compose for consistent local deployment and testing.

Development followed an iterative approach over multiple project phases, with continuous integration of AI training artifacts, Android inference logic, backend services, data model refinements, and dashboard usability improvements. Final deliverables include a trained YOLOv8n model (mAP@50 = 80.52%, mAP@50-95 = 50.29%), Android on-device inference pipeline, implemented API server, database schema/migration, operational dashboard, runtime orchestration files, and this updated technical report. We acknowledge our supervisor and Yarmouk University for technical guidance and academic support throughout the project lifecycle.

Keywords: YOLOv8n, TensorFlow Lite, On-device inference, Pothole management, Geospatial deduplication, ASP.NET Core API

Chapter 1: Introduction

1.1 Problem Statement and Purpose

Urban roads degrade continuously, and potholes may spread faster than maintenance teams can inspect and repair them. Manual inspection workflows are labor-intensive and do not provide continuous, city-scale visibility. This project aims to provide a practical AI-enabled software system that transforms pothole detections into structured operational data, then into actionable maintenance decisions.

1.2 Background

Conventional road-defect management usually relies on periodic field visits and citizen reporting. While both are valuable, they are not sufficient for proactive maintenance planning. A data-driven, location-aware workflow is needed to prioritize repairs and monitor status changes transparently.

1.3 Aims and Objectives

  • Build a secure backend API for pothole ingestion and lifecycle management.
  • Persist pothole records and geospatial attributes in PostgreSQL/PostGIS.
  • Train and evaluate a mobile-ready pothole detection model using YOLOv8n.
  • Deploy TensorFlow Lite inference on Android with real-time camera processing.
  • Implement operational dashboard workflows for map/list/detail management.
  • Support role-based access for admin, operator, and viewer users.
  • Provide export and analytics features for municipal reporting.

1.4 Existing Ways the Problem Is Solved

Existing approaches include manual road surveys, public complaint channels, isolated digital tracking tools, and vision-only research prototypes disconnected from municipal operations workflows. These approaches improve reporting coverage partially, but often miss robust deployment constraints, deduplication, status workflows, and integrated geospatial analytics.

1.5 Main Solution Idea

The main idea is an implementation-centered AI operations platform: detect potholes on-device through a TensorFlow Lite model, convert detections into geo-tagged events, validate and deduplicate them server-side, store them in a geospatial database, then expose map/table/detail workflows for maintenance teams through a secure dashboard.

1.6 Key Technical Details and Evaluation Approach

  • Backend stack: ASP.NET Core 8, Entity Framework Core, JWT, SignalR.
  • Database: PostgreSQL 16 + PostGIS with geography point and indexed access paths.
  • Dashboard: React + Vite + Tailwind + React Leaflet + Axios.
  • AI stack: Ultralytics YOLOv8n trained in Colab and exported as float16 TFLite model.
  • Android inference: model input [1,640,640,3], output [1,5,8400], confidence threshold 0.5.
  • Deduplication: 15-meter radius to avoid duplicate pothole inflation.
  • Evaluation basis: implemented feature completion, workflow correctness, and runtime behavior.

1.7 Contributions

  • Implemented full REST API surface for auth, potholes, users, vehicles, and stats.
  • Implemented geospatial deduplication and confirmation-count behavior in service layer.
  • Prepared and trained AI dataset with 2,642 images and 9,077 annotations for single-class pothole detection.
  • Integrated on-device TensorFlow Lite inference with NNAPI/GPU/CPU delegate fallback on Android.
  • Implemented production-like dashboard workflows (list, map, detail, bulk, export, analytics).

Chapter 3: Design

3.1 Design Overview

3.1.1 System Design to Achieve Solution

The implemented design follows a service-oriented architecture where the backend API acts as the control point for validation, authorization, deduplication, persistence, and analytics exposure. The dashboard is designed for operational decisions, not only visualization. Upstream, an Android AI inference layer performs local pothole detection and emits confidence-scored candidate reports.

3.1.2 How the Design Addresses the Problem Statement

  1. Transforms reports into normalized geospatial records.
  2. Detects potholes directly from camera frames using optimized mobile AI inference.
  3. Prevents duplicate inflation using server-side spatial distance checks.
  4. Provides lifecycle status model for verification and repair tracking.
  5. Supports role-separated actions to reduce operational misuse risk.
  6. Adds export and analytics for planning and external reporting.

3.1.3 Detailed Figure and User Scenarios

Scenario A (Operator): login, filter unverified points, open details, verify or mark false positive, then continue with next priority cluster.

Scenario B (Admin): review trends, bulk mark repaired after maintenance feedback, export CSV for daily reporting.

Legal considerations include lawful handling of geolocation-linked records, controlled account access, and appropriate data retention policies. System design supports policy enforcement through role-based access, API authentication, and centralized backend validation.

  • Infrastructure constraints favored a reproducible local stack before production-scale rollout.

4.8 AI Dataset, Training, and Model Export

AI development was implemented in the ai-model/ workspace using a reproducible Colab workflow and documented scripts/utilities. The final dataset contains 2,642 labeled images and 9,077 pothole instances from Roboflow and Kaggle sources. Training used YOLOv8n with 100 epochs, AdamW optimizer, and augmentation settings tuned for road-scene variation.

All quantitative metrics in this report are anchored to the saved artifacts under ai-model/trained_model/ (especially results.csv and TRAINING_METRICS.md). The trained model was exported to best_float16.tflite and integrated into Android assets. Model I/O specification is fixed to input shape [1, 640, 640, 3] and output shape [1, 5, 8400]. This format supports fast on-device inference and avoids network dependency for core detection functionality.

4.9 Android AI Inference and Reporting Pipeline

The Android inference path is implemented in PotholeDetector.kt and ProcessFrameUseCase.kt. Frames are resized/normalized, inference is executed through TensorFlow Lite with delegate fallback (NNAPI, GPU, CPU), candidate boxes are filtered by confidence, NMS is applied, and resulting detections are passed to reporting logic. Reporting then performs GPS tagging, deduplication checks, and queued upload to the backend.

5.3 Strengths and Weaknesses

Strengths:

  • Documented and reproducible AI training process with dataset and metrics artifacts.
  • Efficient on-device inference architecture with delegate fallback.
  • Clear modular architecture between controllers, services, and data model.
  • Practical geospatial model with index support and dedup behavior.
  • Operationally useful UI (map + list + detail + bulk + export).
  • Containerized startup path suitable for development and staging.

Weaknesses / Current Limitations:

  • Recall can be improved further for safety-critical high-sensitivity deployments.
  • Current AI model supports single-class detection (pothole only).
  • No full audit trail for every status transition and actor action yet.
  • No advanced route optimization for maintenance dispatch in current release.
  • Cloud production hardening and observability still require additional work.

8.2 Further Future Work

  • Add full audit history for status transitions and actor tracking.
  • Improve AI recall through local Jordan data collection and domain adaptation fine-tuning.
  • Extend AI to multi-class road-damage detection and severity estimation.

  • Role models should be implemented early to avoid insecure shortcuts later.
  • Operational UX (bulk actions, filters, exports) matters as much as raw backend completeness.
  • Containerized development improves reproducibility and onboarding speed.

  2. Open Potholes page and apply status/severity filters.
  3. Review unverified/high-confidence records first.
